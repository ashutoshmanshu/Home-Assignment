3 · The order book (the picture in your head)

ASKS (sells, lowest price = best)
1075#00 ·1
1050#03 ·10
1025#05 ·2#07 ·5 ←oldest first
BIDS (buys, highest price = best)
1000#01 ·9#06 ·1
975#02 ·30

Two sorted price ladders.

• Each price level is a queue: oldest order matches first (time priority).

• Best ask = lowest sell. Best bid = highest buy.

• A new order only trades if it crosses the best price on the other side.

4 · The one matching rule

A BUY crosses when buy price ≥ best ask  (a sell: sell price ≤ best bid).

When it crosses → trade for min(both quantities) at the resting order's price.
Still have quantity left? keep eating the next best/oldest order. Nothing left to cross? the remainder rests in the book.

Per pair, output is always: trade → aggressive fill → resting fill. The buy was fully filled, so it never rests.

6 · Why it's fast (the HFT bits)

No allocations

Orders live in one big pre-made array (pool). Links between them are array indexes, not new/delete.

O(1) best price

Price levels are a flat array; a bitmap marks non-empty levels, so best price = one CPU bit-scan, not a tree search.

O(1) cancel

A hash map id → slot jumps straight to the order; unlink it from its queue in O(1).

Lock-free at the edge

Book runs on one thread (fair + deterministic). A lock-free ring only hands work in (--pipeline).
